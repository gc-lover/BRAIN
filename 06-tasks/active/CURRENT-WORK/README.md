# CURRENT-WORK - Структура рабочей документации

**Назначение:** Отслеживание текущей работы над концепциями игры NECPGAME

**Принцип:** Одна фича/одно обсуждение = один файл

---

## Структура директории

```
CURRENT-WORK/
├── README.md                    # Этот файл - описание структуры
├── current-status.md            # Краткий статус текущей работы
├── open-questions.md           # Открытые вопросы для проработки
├── active/                      # Активные обсуждения (в процессе)
│   └── YYYY-MM-DD-feature-name.md
└── archive/                     # Завершенные обсуждения (перемещены после завершения)
    └── YYYY-MM-DD-feature-name.md
```

---

## Работа с файлами

### Создание нового обсуждения

1. Создать файл в `active/` с форматом: `YYYY-MM-DD-feature-name.md`
2. Добавить метаданные:
   ```markdown
   **Статус:** in_progress | completed | archived
   **Приоритет:** high | medium | low
   **Дата создания:** YYYY-MM-DD
   **Дата завершения:** YYYY-MM-DD (если завершено)
   **Связанные документы:** путь к финальному документу
   ```
3. Записывать все обсуждения, вопросы, варианты ответов, решения
4. Когда обсуждение завершено - переместить в `archive/`

### Завершение обсуждения

Когда фича завершена и готова:
1. Обновить статус на `completed`
2. Добавить дату завершения
3. Добавить ссылку на финальный документ (например, `combat-overview.md`)
4. Переместить файл из `active/` в `archive/`

---

## Связь с основным CURRENT-WORK.md

Основной файл `CURRENT-WORK.md` (в родительской директории) теперь является **индексом**:
- Ссылки на активные обсуждения в `active/`
- Краткий статус текущей работы
- Ссылки на завершенные обсуждения в `archive/`

---

## Правила именования файлов

**Формат:** `YYYY-MM-DD-feature-name.md`

**Примеры:**
- `2025-11-02-vision.md`
- `2025-11-02-combat-system.md`
- `2025-11-03-economy-system.md`
- `2025-11-03-hacking-detailed.md`

**Важно:**
- Дата создания обсуждения
- Короткое, понятное имя фичи
- Использовать дефисы, не пробелы
